# Change Assurance 项目背景

## 1. 项目定位

Change Assurance 是一个面向代码变更的证据驱动审查系统。

它服务于两类场景：

1. PR/MR 合并前审查
2. Coding Agent 完成任务后的交付验收

系统目标不是生成更多 Code Review 评论，而是在明确审查范围内，基于可追溯证据，判断一次变更是否具备合并或验收条件。

---

## 2. 问题背景

当前使用 Codex、Claude Code 等 Coding Agent 时，代码审查常依赖模型直接阅读 diff 后输出结论。

这种方式存在几个核心问题：

- 模型可能跳过关键审查步骤，只关注局部代码细节。
- 模型容易把推测、偏好或不完整上下文包装成确定问题。
- 模型可能声称测试已通过、命令已执行，但缺少真实执行证据。
- Review 输出通常缺少覆盖范围，无法判断哪些风险面未被审查。
- 不同模型、不同 Skill、不同提示词之间容易出现审查标准漂移。
- Coding Agent 的“任务已完成”通常只是自述，缺少独立验收机制。

因此，单纯依赖 Prompt 或 Skill，无法为 Code Review 提供稳定、可回放、可验证的质量保障。

---

## 3. 核心目标

Change Assurance 希望建立一套可复用的变更审查能力，使审查结论具备以下特征：

- **证据驱动**：每个阻断性结论都必须关联代码、命令输出、测试结果或明确需求。
- **过程可控**：审查按固定阶段进行，避免模型自由跳步。
- **结论可审计**：能够追溯输入、执行命令、审查产物、问题来源和最终判断。
- **模型可替换**：Codex 与 Claude Code 只是推理执行器，不定义审查标准。
- **仓库可适配**：通用审查理论与仓库特定规则分离。
- **持续可评测**：通过固定案例和回归评测，验证审查能力是否真实提升。

---

## 4. 核心原则

### 4.1 审查目标不是“多找问题”

审查的目标是判断变更是否：

- 引入不可接受的回归或风险；
- 降低整体代码健康度；
- 未完成声明的需求或验收条件；
- 缺少足以支持合并或验收的证据。

### 4.2 事实、推断与假设必须分层

所有审查结论必须区分：

- **Observed**：代码、diff、命令输出、测试结果中直接观察到的事实。
- **Derived**：基于多个事实推导出的结论。
- **Hypothesis**：合理怀疑但尚未被充分验证的风险。

未经证实的假设不得直接作为阻断性结论。

### 4.3 Blocker 必须具备完整证据链

一个阻断问题至少应包含：

- 代码或行为定位；
- 触发条件；
- 行为偏差；
- 用户或系统影响；
- 可追溯证据；
- 明确修复要求。

### 4.4 审查覆盖范围必须显式记录

任何“建议合并”或“验收通过”都必须同时说明：

- 已审查范围；
- 已执行验证；
- 未执行验证；
- 未覆盖风险；
- 需要人工确认的边界。

---

## 5. 目标边界

### 本项目要做

- 提供通用的 Code Review Skill 理论与审查协议。
- 提供 Harness 设施，冻结输入、执行受控验证、保存审查产物。
- 提供结构化 artifact，例如 evidence ledger、issue ledger、coverage ledger。
- 提供 Codex 与 Claude Code 的接入适配层。
- 支持 PR/MR 合并前审查的 dry run。
- 后续支持 Coding Agent 的任务交付验收。

### 本项目暂不做

- 自动批准、拒绝或合并 PR/MR。
- 自动修改业务代码或自动修复问题。
- 替代人类在产品、架构或业务决策上的责任。
- 试图证明代码“绝对正确”。
- 在第一阶段构建复杂的多 Agent 平台或完整 CI 审批系统。

---

## 6. 成功标准

第一阶段成功，不以“发现问题数量”衡量，而以以下能力衡量：

- 能稳定生成带证据的审查报告。
- 不会把未执行命令描述为已通过验证。
- 不会把无证据猜测升级为 blocker。
- 能明确输出审查覆盖范围与未覆盖范围。
- 对相同输入，能够在不同模型运行时保持接近的审查协议与输出结构。
- 能通过固定 golden cases 验证问题发现、严重等级和证据完整性。

---

## 7. 一句话定义

> Change Assurance 是一个以 Skill 提供审查判断能力、以 Harness 提供可信执行与证据约束的变更质量保障系统。
